Composition over the inheritance

Posted on 2019-09-20 Edited on 2019-11-11 Disqus:

The Story

I recently encountered a question about why it’s better to using interface than inheritance. I have this because we defined lots of interfaces (behaviors) in a recent project. And, I am not saying this is bad, but just because this is not my familiar pattern when trying to design a system and also wanted to think of it deeper why we are doing that. I am confused if some classes share the same behaviors, my first thought from Java world is to create a parent class to define those common behaviors and then derivative classes would get the benefits without implementing the same behaviors again. But someone told me: Using an interface is better. So, I want to figure out why an interface is better than inheritance. According to the wiki description:

Composition over inheritance - Wikipedia
Composition over inheritance (or composite reuse principle) in object-oriented programming (OOP) is the principle that classes should achieve polymorphic behavior and code reuse by their Composition (by containing instances of other classes that implement the desired functionality) rather than inheritance from a base or parent class

Use ESLint and Prettier to format your code in TypeScript

Posted on 2019-08-10 Edited on 2019-10-30 Disqus:

Use ESLint and Prettier to format your code in TypeScript

I think ESLint[1] is an essential thing when you try to create a project using JavaScript/TypeScript. That would be an excellent tool to parse the potential issues in your code.

Let’s check the philosophy of ESLint:

  • Everything is pluggable:
    • Rule API is used both by bundled and custom rules
    • Formatter API is used both by bundled and custom formatters
    • Additional rules and formatters can apply at runtime
    • Rules and formatters don’t have to be bundled to combine
  • Every rule:
    • Is standalone
    • Can be turned off or on (nothing can be deemed “too important to turn off”)
    • Can be set to a warning or error individually
  • Additionally:
    • Rules are “agenda-free” - ESLint does not promote any particular coding style
    • Any bundled rules are generalizable
  • The project:
    • Values documentation and clear communication
    • Is as transparent as possible
    • Believes in the importance of testing

Japan Town - The Northern California Cherry Blossom Festival

Posted on 2019-04-22 Edited on 2019-11-12 Disqus:

日本城櫻花節 (Japan Town - The Northern California Cherry Blossom Festival)

每年四月第三個週末很熱鬧, 正好是美國的復活節 Easter週末, 在這周前的一兩週各地的活動很多, 我們也趁熱鬧跑去 Golden Gate Park 參加了撿彩蛋的活動, 不過兒子不太買單, 覺得撿彩蛋頗無聊, 倒是我撿到快累死了.

所以星期六 (4/20) 想說不然來去日本城, 那邊聽說有櫻花節活動. 一到現場一看不得了啊,完全是人山人海, 整個道路都封街了:


Nor Cal Cherry Blossom Festival

這封街的範圍有點大. 當時我開車走 Geary Blvd 完全找不到停車位. 後來還是停到 日本城裡面的停車場, 畢竟車子停外面風險太大, 真心傷不起. 只好花點小錢看能不能好好保護一下, 殊不知我想得太天真: 不管到哪裡風險都存在, 問題在於你運氣夠不夠好.

後來上網一查才知道, 原來日本城的櫻花節算是歷史悠久, 從1968年就開始了. 算是北加州日裔族群大的歡慶活動, 可以參考: History – Northern California Cherry Blossom Festival. 這個歡樂節慶其實有一整週的時間, 不過我們好趕上週末人最多的時候, 當天擺攤的攤位每攤基本上都要排隊, 對於排隊沒有任何興趣的我, 最後還是排了日本城裡面最富盛名的 Matcha Cafe Maiko, 這家實在太有名了. 基本上我十分推薦他們家的 Matcha Ice Cream. 抹茶味非常濃厚, 但不會讓你覺得很膩, 又不死甜, 吃了基本上就是會升天那種 (可惜食尚玩家不玩了, 不然我也去應徵外景主持人). 另外, 他們有金箔口味的將軍, 先來看一下圖:

一大片金箔, 是不是很威猛, 可惜我是從來沒點過, 但聽吃過的朋友都讚不絕口, 以後有機會過來的吃吃的話, 請大家一定不能錯過.

吃完冰淇淋之後我其實一直在想要不要衝二樓的拉麵, 這家真心不騙非常好吃, Yelp 高達 4.7顆星評價, 只是真的人永遠這麼多, 完全排不到, 後來想想還是放棄, 去廣場看一下活動好了.

我兒子受到台上的感召也是在旁邊又唱要跳的, 看來一代舞王的養成要準備開始了. KrKr.

後續:

原本想說停到室內停車場比較有保障, 結果到繳費機就看到兩位老先先老太太在他們新車車頭那邊不知道在討論什麼. 一看不得了, Lexus IS 300 全新車, 車頭被刮了!!!!! 立馬手刀回自己的車子查看. 還好還好, 我停得夠角落, 夠低調, 沒什麼人注意到, 只能幫他們 QQ了. 舊金山最近 Window Break-in 越來越多, 還有人會故意去刮新車, 看來得想個辦法了, 囧.
